## Summary
The user asked for "app key and apikey installation so unique like openalgo". Openalgo and dittot generate per-install secrets via `openssl rand -hex 32` and store them in `.env`. Watify needs the same: a per-install cryptographic identity that survives re-runs of the installer.

Two distinct keys (deliberately separate so rotating one does not invalidate the other):
- **`WATIFY_APP_SECRET`** -- the server-side signing key. Used for JWT HMAC (TKT-0024), cookie HMAC (if/when we sign cookies), CSRF token signing, anywhere we need a server-only secret.
- **`WATIFY_API_KEY`** -- a per-install bearer key for programmatic clients (future webhook receivers, the eventual Watify SDK, automation scripts).

These are SEPARATE from:
- `WATIFY_SESSION_ENCRYPTION_KEY` (Fernet, wars session blob, TKT-0011).
- The argon2 password hash (per-user, never an env var).

## Expected
### settings.py
```python
# Per-install identity (TKT-0031). Generated by install.sh on fresh
# setup; preserved on re-runs. Application code that needs server
# signing reads app_secret; programmatic Bearer auth reads api_key.
app_secret: str = ""   # required for auth endpoints (TKT-0024)
api_key:    str = ""   # optional in v1 (no Bearer endpoint yet)
```
Auth router raises `503 auth_not_configured` if `app_secret` is empty.

### install/install.sh (folds into TKT-0030)
```bash
if grep -q '^WATIFY_APP_SECRET=' "$ENV_FILE" 2>/dev/null; then
    log_info "Preserving existing WATIFY_APP_SECRET"
else
    APP_SECRET=$(openssl rand -hex 32)
    echo "WATIFY_APP_SECRET=$APP_SECRET" >> "$ENV_FILE"
fi
# Same pattern for WATIFY_API_KEY.
```
On reinstall the script detects existing values and preserves them. Re-pair, re-deploy, etc. do not invalidate sessions.

### Backend boot guard
On lifespan startup, log `WARN: WATIFY_APP_SECRET is empty -- /api/auth/* will return 503` if unset. Do not crash the whole app (other features still work).

### /api/auth/me extension (later, with TKT-0024)
- Authed user with the right `Authorization: Bearer <api_key>` header (in addition to cookie) can also call /api/auth/me. This is the bridge for the eventual Watify SDK.

### Backend identity dump for diagnostics
A small `app/identity.py`:
```python
def fingerprint() -> str:
    """First 8 hex chars of sha256(app_secret) -- safe to log/expose
    so the operator can confirm two boxes share the same secret
    without revealing it."""
```
Hit at `GET /api/health` -> include `{"app_fingerprint": "a1b2c3d4"}`.

### backend/.env.example
Document both vars + the `openssl rand -hex 32` snippet.

## Acceptance
- Fresh install -> .env has both vars, each 64 hex chars.
- Re-install -> same values preserved.
- Two installs on different hosts -> different fingerprints in /api/health.
- Auth endpoints (when TKT-0024 ships) refuse to operate without `app_secret`.
